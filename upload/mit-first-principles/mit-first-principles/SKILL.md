# MIT First Principles Framework

Three battle-tested principles from MIT's engineering culture for strategic thinking, prioritization, and breakthrough problem-solving.

## When to Use

Trigger this skill when users:
- Face overwhelming complexity and need prioritization frameworks
- Want to understand a system deeply before acting
- Need to separate facts from assumptions
- Ask for first principles thinking or problem decomposition
- Feel overwhelmed by too many tasks or projects
- Want to test assumptions systematically
- Need frameworks for strategic decision-making
- Ask "where do I even start?" with complex problems
- Want to balance creativity with rigor
- Need to distinguish valuable work from busy work

## Core Philosophy

MIT culture produces breakthrough innovation through three interconnected principles:

**Principle 1**: Master the System → Understand deeply before acting
**Principle 2**: Pass the Fire Hose Test → Focus on what truly matters
**Principle 3**: Think Like a Founder → Decompose problems to fundamentals

Each principle builds on the others: mastery enables discernment, discernment enables focus, focus enables breakthrough.

---

## Principle 1: Master the System (Hacker Mindset)

**Core Insight**: Act like a hacker, not a hammer.

When facing a complex system—whether a codebase, organization, market, or technology—resist the urge to force solutions. Instead, develop deep understanding first.

### The MIT Hack Standard

MIT "hacks" (like placing a police car on the Great Dome) require:
- Total precision in understanding the system
- Knowledge so deep you can play with it safely
- Creativity that emerges from mastery, not ignorance

### Application Framework

```
SYSTEM MASTERY CHECKLIST

Before acting on any complex system:

□ OBSERVE: What does the system actually do? (not what it should do)
□ MAP: What are the key components and their relationships?
□ TEST: What happens when I change small things?
□ LIMITS: Where are the boundaries and constraints?
□ PLAY: Can I experiment safely before committing?

MASTERY INDICATORS:
• Can I explain how it works to a novice?
• Can I predict what will happen if X changes?
• Do I understand WHY it was built this way?
• Have I found at least one non-obvious insight?
```

### Anti-Pattern Detection

| Hammer Behavior | Hacker Behavior |
|-----------------|-----------------|
| Rush to implement | Invest time understanding |
| Force familiar solutions | Discover system-native approaches |
| Work harder when stuck | Step back and observe more |
| Ignore unexpected results | Investigate anomalies curiously |
| Treat stress as normal | Treat confusion as a signal |

### Key Principle

**Smart doesn't have to be stressful.** If you're forcing solutions, you haven't understood the system deeply enough. Mastery enables elegance.

## Principle 2: Pass the Fire Hose Test (Focused Prioritization)

**Core Insight**: The fire hose was never meant to be drunk from.

MIT's overwhelming workload teaches a crucial lesson: it is mathematically impossible to do everything well. The test isn't about capacity—it's about learning to choose.

### The 3I Model

Before committing to any project or task, apply three filters:

```
THE 3I PRIORITIZATION MODEL

┌─────────────────────────────────────────────────────────────────┐
│                           FILTER 1                               │
│                        IS IT IMPORTANT?                          │
│                                                                  │
│  "Will this matter a year from now?"                             │
│                                                                  │
│  □ YES → Continue to Filter 2                                    │
│  □ NO  → Deprioritize or delegate                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                           FILTER 2                               │
│                        IS IT IMPACTFUL?                          │
│                                                                  │
│  "Does it actually move the needle, or is it busy work?"         │
│                                                                  │
│  □ YES → Continue to Filter 3                                    │
│  □ NO  → Question necessity                                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                           FILTER 3                               │
│                        IS IT IRREVERSIBLE?                       │
│                                                                  │
│  "If it breaks, can it be fixed later?"                          │
│                                                                  │
│  □ YES → Highest priority—do carefully                           │
│  □ NO  → Can be deferred or done quickly                         │
└─────────────────────────────────────────────────────────────────┘
```

### 3I Decision Matrix

| Combination | Priority | Action |
|-------------|----------|--------|
| Important + Impactful + Irreversible | CRITICAL | Do first, do carefully |
| Important + Impactful + Reversible | HIGH | Schedule soon, acceptable risk |
| Important + Low Impact + Irreversible | MEDIUM | Careful but lower priority |
| Important + Low Impact + Reversible | LOW | Batch or delegate |
| Not Important | DECLINE | "No" is a complete sentence |

### Fire Hose Analysis Template

```
FIRE HOSE AUDIT: [Context/Role/Project]

Date: [Date]
Current overwhelm level: [1-10]

LIST EVERYTHING DEMANDING ATTENTION:
(Don't filter—just dump)

1. [Task/Project]
2. [Task/Project]
3. [Task/Project]
...

NOW APPLY 3I TO EACH:

| Item | Important? | Impactful? | Irreversible? | Priority | Decision |
|------|------------|------------|---------------|----------|----------|
| #1   | Y/N        | Y/N        | Y/N           | C/H/M/L  | [Action] |
| #2   | Y/N        | Y/N        | Y/N           | C/H/M/L  | [Action] |
| #3   | Y/N        | Y/N        | Y/N           | C/H/M/L  | [Action] |

KEEP (Critical/High):
- [Item]
- [Item]

DEFER (Medium/Low):
- [Item]
- [Item]

DROP/DELEGATE (Not Important):
- [Item]
- [Item]

REALIZATIONS:
[What patterns emerge? What should you say no to more often?]
```

### Key Principle

**Choose between valuable and more valuable.** The fire hose forces you to distinguish what matters from what merely seems urgent.

## Principle 3: Think Like a Founder (First Principles Decomposition)

**Core Insight**: Question assumptions, rebuild from fundamentals.

Instead of asking "How do I solve this problem?", ask:
- What do I truly know about this?
- What am I assuming?
- What pieces are still missing?

### The Elon Musk Example

Conventional thinking: "Rockets cost $65 million."

First principles analysis:
- What are rockets actually made of? (Aluminum, titanium, carbon fiber, etc.)
- What do those raw materials cost? (~2% of rocket price)
- Why the 50x markup? (Supply chain, regulations, legacy processes)

Result: SpaceX builds rockets for a fraction of incumbent costs.

### First Principles Decomposition Framework

```
FIRST PRINCIPLES BREAKDOWN

STEP 1: WRITE THE PROBLEM
[State the complex problem in 1-3 clear sentences]

_________________________________________________________________________

STEP 2: THREE-COLUMN ANALYSIS

┌────────────────────┬────────────────────┬────────────────────┐
│       FACTS        │    ASSUMPTIONS     │     NEXT STEPS     │
│   (What I KNOW)    │  (What I BELIEVE)  │   (What to TEST)   │
├────────────────────┼────────────────────┼────────────────────┤
│                    │                    │                    │
│ •                  │ •                  │ •                  │
│                    │                    │                    │
│ •                  │ •                  │ •                  │
│                    │                    │                    │
│ •                  │ •                  │ •                  │
│                    │                    │                    │
│ •                  │ •                  │ •                  │
│                    │                    │                    │
└────────────────────┴────────────────────┴────────────────────┘

STEP 3: ASSUMPTION SELECTION
Which assumption, if wrong, would change everything?
→ [Selected assumption to test this week]

STEP 4: TEST DESIGN
How will I test this assumption?
• Method: [How to test]
• Data needed: [What evidence would confirm/refute]
• Timeline: [This week]
• Success criteria: [How I'll know the answer]
```

### Fact vs. Assumption Distinguisher

Use these questions to categorize each belief:

| Question | If YES | If NO |
|----------|--------|-------|
| Can I point to direct evidence? | FACT | Likely assumption |
| Have I personally verified this? | FACT | Likely assumption |
| Is this based on someone else's opinion? | ASSUMPTION | Possibly fact |
| Does "everyone know" this? | ASSUMPTION | Investigate |
| Am I extrapolating from limited data? | ASSUMPTION | Need more facts |

```
[The one assumption that, if wrong, changes everything]

• By when: [Date]

```

### Key Principle

**Deconstruct before you construct.** Breaking problems into Facts, Assumptions, and Next Steps transforms how you think about all problems.

## Integrated Framework: Three Principles Together

Apply the three principles in sequence for maximum impact:

```
MIT FIRST PRINCIPLES: INTEGRATED APPROACH

PHASE 1: MASTER THE SYSTEM
┌─────────────────────────────────────────────────────────────┐
│ Before doing anything:                                       │
│ • Observe the system without judgment                        │
│ • Map key components and relationships                       │
│ • Find the non-obvious insights                              │
│ • Earn the right to have an opinion                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
PHASE 2: PASS THE FIRE HOSE TEST
┌─────────────────────────────────────────────────────────────┐
│ Once you understand the system:                              │
│ • List all potential actions/projects                        │
│ • Apply 3I filter (Important, Impactful, Irreversible)       │
│ • Ruthlessly prioritize                                      │
│ • Say no to valuable-but-not-most-valuable                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
PHASE 3: THINK LIKE A FOUNDER
┌─────────────────────────────────────────────────────────────┐
│ For your highest-priority challenge:                         │
│ • Separate Facts from Assumptions                            │
│ • Identify the highest-leverage assumption                   │
│ • Design a test for this week                                │
│ • Build from fundamentals, not conventions                   │
└─────────────────────────────────────────────────────────────┘
```

## Quick Reference: When to Use Each Principle

| Situation | Lead With | Why |
|-----------|-----------|-----|
| New job, team, or project | Principle 1 (Master) | Understand before acting |
| Feeling overwhelmed | Principle 2 (Fire Hose) | Prioritize ruthlessly |
| Stuck on hard problem | Principle 3 (First Principles) | Question assumptions |
| Everything seems urgent | Principle 2 (Fire Hose) | Not everything is important |
| Standard approaches failing | Principle 3 (First Principles) | Conventional wisdom may be wrong |
| Making costly mistakes | Principle 1 (Master) | You haven't understood deeply enough |

## Common Scenarios

### Scenario 1: Starting a New Complex Project

**Framework Sequence**:
1. **Master the System**: Spend dedicated time understanding the domain, stakeholders, and constraints before proposing solutions
2. **Fire Hose Test**: Identify which aspects of the project are truly critical vs. nice-to-have
3. **First Principles**: For the critical aspects, decompose into facts vs. assumptions and test key assumptions early

### Scenario 2: Drowning in Priorities

**Framework Sequence**:
1. **Fire Hose Test**: Dump everything, apply 3I, create clear tiers
2. **First Principles**: For top-tier items, question whether you're solving the right problem
3. **Master the System**: For items you keep, ensure you understand them deeply enough to solve elegantly

### Scenario 3: Breaking Through on a Stuck Problem

**Framework Sequence**:
1. **First Principles**: Separate what you know from what you assume
2. **Master the System**: Investigate the areas where your assumptions might be wrong
3. **Fire Hose Test**: Focus your investigation on the most leveraged assumptions
