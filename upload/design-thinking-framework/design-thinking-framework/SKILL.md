# Design Thinking Framework

Human-centered innovation methodology for solving complex problems through empathy, experimentation, and iteration.

## When to Use

Trigger this skill when users:
- Request design thinking workshops or facilitation guides
- Need user research methods (interviews, observations, journey maps)
- Want problem reframing or "How Might We" statements
- Ask for ideation techniques (brainstorming, Crazy 8s, SCAMPER)
- Need prototyping approaches (low-fi, rapid, paper prototypes)
- Request user testing or feedback synthesis methods
- Want customer journey mapping or experience design
- Need service design or service blueprinting
- Ask for empathy maps or persona development
- Request innovation frameworks for product development
- Need to shift from solution-first to problem-first thinking

## Core Philosophy

Design thinking inverts traditional problem-solving: instead of starting with solutions, start with people. Three pillars define the approach:

**Desirability** (Human): What do people need?
**Feasibility** (Technical): What is technically possible?
**Viability** (Business): What is financially sustainable?

Innovation lives at the intersection of all three.

## The 5-Phase Process

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  EMPATHIZE  │ →  │   DEFINE    │ →  │   IDEATE    │ →  │  PROTOTYPE  │ →  │    TEST     │
│             │    │             │    │             │    │             │    │             │
│ Understand  │    │ Synthesize  │    │ Generate    │    │ Build to    │    │ Learn from  │
│ users       │    │ insights    │    │ solutions   │    │ think       │    │ users       │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                           ↑                                      │
                                           └──────────────────────────────────────┘
                                                        (Iterate)
```

**Key Principle**: The process is non-linear. Loop back based on learnings.

## Phase 1: EMPATHIZE

**Goal**: Develop deep understanding of users, their needs, behaviors, and pain points.

### Tool 1.1: User Interview Guide

**Purpose**: Structured conversations to uncover needs, motivations, and frustrations.

**Template**:
```
USER INTERVIEW GUIDE: [Research Focus]

INTERVIEW METADATA
- Participant: [Pseudonym/ID]
- Date: [Date]
- Duration: [Time]
- Context: [Where/how interviewed]

WARM-UP (5 min)
- Tell me about yourself and your role.
- Walk me through a typical day.

MAIN QUESTIONS (30-40 min)

Context Questions:
1. Tell me about the last time you [did activity related to problem].
2. What were you trying to accomplish?
3. Walk me through exactly what you did.

Emotion Questions:
4. How did that make you feel?
5. What was the hardest part?
6. What would have made it better?

Comparison Questions:
7. How does this compare to how you've done it before?
8. What alternatives have you tried?
9. Why did you choose this approach?

Magic Wand Question:
10. If you could wave a magic wand and change anything about [experience], what would it be?

PROBING TECHNIQUES
- "Tell me more about that..."
- "Why do you think that is?"
- "What happened next?"
- "Can you give me an example?"
- [5 seconds of silence]

OBSERVATION NOTES
- Body language: [Notes]
- Emotions shown: [Notes]
- Surprises: [Notes]
- Quotes to remember: [Verbatim quotes]

KEY TAKEAWAYS
1. [Main insight]
2. [Secondary insight]
3. [Unexpected finding]
```

## Phase 2: DEFINE

**Goal**: Synthesize research into actionable problem statements.

### Tool 2.1: Point of View (POV) Statement

**Purpose**: Frame the problem from the user's perspective.

**Template**:
```
POINT OF VIEW STATEMENT

[USER] needs [NEED] because [INSIGHT].

COMPONENTS:

USER (specific, not generic):
- Not: "A customer"
- Yes: "A busy working parent with two kids under 5"

NEED (verb, not solution):
- Not: "needs an app"
- Yes: "needs to coordinate family schedules efficiently"

INSIGHT (surprising, based on research):
- Not: "because they're busy"
- Yes: "because the mental load of remembering everyone's activities creates constant anxiety"

FULL POV:
"A busy working parent with two kids under 5 needs to coordinate family schedules efficiently because the mental load of remembering everyone's activities creates constant anxiety, not the actual scheduling itself."

VALIDATION CHECKLIST:
☐ Specific user (not generic "users")
☐ Need is a verb (not a solution)
☐ Insight is surprising (not obvious)
☐ Creates tension that demands resolution
☐ Opens space for multiple solutions
```

### Tool 2.2: How Might We (HMW) Statements

**Purpose**: Reframe problems as opportunities for ideation.

**Template**:
```
HOW MIGHT WE STATEMENTS

POV: [Your Point of View statement]

GENERATING HMWs:

1. Amp Up the Good
   HMW amplify [positive aspect of experience]?
   → HMW [specific amplification]?

2. Remove the Bad
   HMW eliminate [pain point]?
   → HMW [specific elimination]?

3. Explore the Opposite
   HMW make [task] something users look forward to?
   → HMW [specific inversion]?

4. Question Assumptions
   HMW [do thing] without [assumption]?
   → HMW [specific challenge]?

5. Adjust Time
   HMW make [task] happen before/after it normally does?
   → HMW [specific timing change]?

6. Change the Status Quo
   HMW completely reimagine [current approach]?
   → HMW [specific reimagining]?

7. Break It into Pieces
   HMW address just the [specific part] of [problem]?
   → HMW [specific piece]?

SELECTED HMWs FOR IDEATION:

Primary (most promising):
- HMW [statement 1]?

Secondary (good alternatives):
- HMW [statement 2]?
- HMW [statement 3]?

HMW QUALITY CHECK:
☐ Not too broad (can't ideate) - "HMW make users happy?"
☐ Not too narrow (solution embedded) - "HMW add a calendar app?"
☐ Inspiring (generates excitement)
☐ Based on real insights (not assumptions)
```

## Phase 5: TEST

**Goal**: Learn from users to refine solutions.

### Tool 5.1: User Testing Protocol

**Purpose**: Structured approach to gathering prototype feedback.

**Template**:
```
USER TESTING PROTOCOL

PROTOTYPE: [Name]
TEST DATE: [Date]
PARTICIPANTS: [N users, recruitment criteria]

PRE-TEST SETUP
- Prototype ready and reset
- Recording setup (if applicable, with consent)
- Quiet space arranged
- Notetaker assigned

INTRODUCTION SCRIPT (2 min)
"Thanks for helping us today. We're testing a [prototype], not you. There are no wrong answers. Please think aloud as you go. We may stay quiet to observe—that's intentional. Ready?"

BACKGROUND QUESTIONS (3 min)
1. Tell me about how you currently [relevant activity].
2. What's most frustrating about that?

TASK SCENARIOS (15-20 min)

Task 1: [Name]
"Imagine you want to [goal]. Using this prototype, please show me how you would do that."
- Success criteria: [What constitutes completion]
- Watch for: [Specific behaviors]

Task 2: [Name]
"Now suppose you want to [second goal]..."
- Success criteria: [What constitutes completion]
- Watch for: [Specific behaviors]

Task 3: [Name]
"Finally, please [third task]..."
- Success criteria: [What constitutes completion]
- Watch for: [Specific behaviors]

FOLLOW-UP QUESTIONS (5 min)
1. What stood out most to you?
2. What was confusing?
3. What would you change?
4. How likely would you use something like this? Why?
5. What's missing?

OBSERVATION CAPTURE:
| Task | Completed? | Time | Errors | Quotes | Body Language |
|------|------------|------|--------|--------|---------------|
| 1    | Y/N        | [s]  | [#]    | "[Q]"  | [Notes]       |
| 2    | Y/N        | [s]  | [#]    | "[Q]"  | [Notes]       |
| 3    | Y/N        | [s]  | [#]    | "[Q]"  | [Notes]       |
```

## Workshop Facilitation Guide

### Design Sprint Structure (5 days)

```
DAY 1: UNDERSTAND (Empathize + Define)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Morning: Expert interviews, user research review
Afternoon: Journey mapping, problem framing
Output: POV statement, HMW questions

DAY 2: SKETCH (Ideate)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Morning: Lightning demos (inspiration), Crazy 8s
Afternoon: Solution sketching, concept development
Output: Individual solution sketches

DAY 3: DECIDE (Ideate → Prototype)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Morning: Critique, voting, decision-making
Afternoon: Storyboarding winning concept
Output: Detailed storyboard for prototype

DAY 4: PROTOTYPE (Prototype)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Full day: Build realistic prototype
Output: Testable prototype

DAY 5: TEST (Test)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Full day: 5 user interviews with prototype
Output: Validated learnings, next steps
```

### Half-Day Workshop Structure (4 hours)

```
HOUR 1: Empathize
- Research share-out (15 min)
- Empathy mapping (30 min)
- Journey mapping (15 min)

HOUR 2: Define
- POV development (20 min)
- HMW generation (20 min)
- HMW voting and selection (20 min)

HOUR 3: Ideate
- Crazy 8s (15 min)
- Concept development (25 min)
- Share and critique (20 min)

HOUR 4: Plan
- Idea selection (15 min)
- Prototype planning (25 min)
- Next steps and owners (20 min)
```

## Common Anti-Patterns to Avoid

| Anti-Pattern | Problem | Fix |
|--------------|---------|-----|
| Solution-first | Jumping to features before understanding users | Mandate empathy phase before ideation |
| Groupthink ideation | Everyone converges on first idea | Silent generation before discussion (Crazy 8s) |
| Precious prototypes | Over-investing before validation | Time-box prototype builds, lower fidelity |
| Leading questions | Biasing user feedback | Use "show me" tasks, not "would you like" questions |
| Confirmation bias | Seeking validation, not truth | Assign "devil's advocate" role, test with skeptics |
| One-and-done | Single test, no iteration | Plan 2-3 test cycles minimum |
| HiPPO decisions | Highest-paid person decides | Use voting with dot stickers, blind selection |

---

## Quick Reference: Phase Outputs

| Phase | Key Output | Quality Check |
|-------|------------|---------------|
| Empathize | Empathy maps, journey maps, interview notes | Based on real user contact (not assumptions) |
| Define | POV statement, HMW questions, personas | Specific user, verb need, surprising insight |
| Ideate | 50+ ideas, concept sketches, selection matrix | Quantity generated before quality judged |
| Prototype | Testable artifact, test plan | Matches fidelity to learning goal |
| Test | Synthesis grid, validated learnings | Clear go/pivot/kill decision |
